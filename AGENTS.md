# AGENTS.md

> このファイルは **AIエージェント（Claude Code, Cursor, GitHub Copilot, Codex, Cline 等）** がこのテンプレートをあなたのアプリ用にカスタマイズするための **正式な指示書** です。
>
> このリポジトリで作業するAIは、まずこのファイルを読んでから編集を始めてください。

---

## 1. このプロジェクトの目的

これは、**生成AIで個人開発したモバイル/Webアプリを世に出すための、ランディングページ（LP）テンプレート**です。技術スタックは Next.js (App Router) + TypeScript + Tailwind CSS の JAMstack 構成で、Vercel への即時デプロイを想定しています。

利用者（あなたに指示を出している開発者）は、AIエージェント（あなた）にこのテンプレートをカスタマイズしてもらい、自分のアプリ専用のLPを最短で公開することを望んでいます。

---

## 2. アーキテクチャの大原則

このテンプレートは「**1ファイルを編集すれば全部変わる**」という設計思想で作られています。コンポーネントを書き換える前に、まず以下を理解してください。

### 編集する場所

| 編集したいもの | 編集するファイル |
| --- | --- |
| アプリ名、説明、価格、機能、FAQなど **テキストや構造** | `site.config.ts` |
| 配色（メイン・アクセント・背景） | `app/globals.css` の CSS変数（`:root` 内） |
| プライバシーポリシー本文 | `content/privacy.md` |
| 利用規約本文 | `content/terms.md` |
| トップページのセクション順序・追加・削除 | `app/page.tsx` |

### 編集してはいけない / 慎重にすべき場所

- `components/` 配下のコンポーネントは、**まず `site.config.ts` のスキーマで表現できないか検討してください**。コンポーネントを直接編集するのは、利用者が明示的にレイアウトの変更を求めた場合のみにしてください。
- `lib/markdown.ts` の置換ロジックを変えるときは、`content/*.md` 内のプレースホルダ（`{appName}` など）との整合性に注意してください。

## 3. 標準カスタマイズ手順

利用者から「**私のアプリ用にカスタマイズして**」と指示された場合、以下の順序で進めてください。

### Step 1. ヒアリング（情報が不足している場合のみ）

最低限以下を確認してください。**すでに指示文に含まれている場合は再度聞かないでください。**

- アプリ名と一言説明（何をするアプリか）
- ターゲットユーザー
- 主要機能（3つ程度）
- 料金プラン（無料のみ？有料あり？金額は？）
- iOS/Android/Web のどれに対応か
- 運営者名（個人名 or 屋号）と連絡用メールアドレス
- 配色の好み（任意）

### Step 2. `site.config.ts` を書き換える

このファイルがサイト全体の中核です。トップから順に、以下の各ブロックを書き換えてください。

1. **`app`**: `name`, `tagline`, `description`, `seoDescription`, `url` を更新
2. **`brand`**: 配色トーンの希望に応じて `mode` (`light` / `dark`) と `heroStyle` を選ぶ
3. **`store`**: ストアURL。リリース前なら空文字のままで構いません（CTA部分が自動で説明文に切り替わります）
4. **`nav`**, **`footer`**: 通常はそのまま。ページを増減した場合はリンクを調整
5. **`hero`**: ファーストビュー。`headline` は配列で改行を表現。最後の要素がイタリック表示になる仕様です
6. **`features`**: 3つの主要機能。アイコンは絵文字や記号（`✦`, `◈`, `✺` 等）で十分。利用者が好みを指定しなければ、シンプルな幾何学記号で揃えてください
7. **`steps`**: 「ダウンロード → セットアップ → 利用開始」の3ステップ。Webアプリなら「アクセス → 登録 → 利用開始」に
8. **`pricing.plans`**: 利用者の指定に応じて。**有料プランがない場合、配列を1要素（Freeのみ）にしても2要素にしてもOK**。`highlighted: true` は1つだけにすること
9. **`faq.categories`**: 最低3カテゴリ、各2問以上を目安に。利用者のアプリの特性に合わせて、想定される質問を生成してください
10. **`contact`**: `email` は必須。`endpoint` は空文字でOK（mailto: にフォールバックします）
11. **`legal`**: `operatorName`、`operatorType`、`lastUpdated` は必ず更新。住所は個人開発者なら空文字で構いません

### Step 3. 配色を調整する（必要に応じて）

`app/globals.css` の `:root` ブロックを編集します。**変更するのは色のみ**、変数名は変えないでください。

```css
:root {
  --color-brand: #0a0a0a;       /* メインカラー（ボタン・見出し背景） */
  --color-brand-fg: #ffffff;    /* メインカラー上の文字色 */
  --color-accent: #ff5b39;      /* アクセント色（ハイライト・リンク等） */
  /* ... */
}
```

利用者がムードを指定したとき（例：「Notionっぽく」「ミニマルで」「ポップに」）の判断指針：

- **ミニマル / 高級感**: `brand` を黒〜濃グレー、`accent` を控えめな色（深い緑、ボルドー、ネイビー）
- **ポップ / カジュアル**: `brand` を鮮やかな色、`accent` を補色で対比
- **ダーク前提**: `brand.mode = 'dark'` にして、`globals.css` の `.theme-dark` ブロックを微調整
- **テック系**: `heroStyle: 'grid'` + モノクロ寄り
- **クリエイティブ系**: `heroStyle: 'aurora'` or `'mesh'` + 鮮やかな色

**禁則**: `Inter`、`Roboto` への変更、紫→ピンクのグラデーション、「AI生成感」のあるクリシェなテーマは避けてください。このテンプレートは Fraunces（ディスプレイ）+ Geist（本文）でエディトリアルな雰囲気を意図しています。利用者が明示的に変えてほしいと言わない限り、フォントは触らないでください。

### Step 4. 法的ページのプレースホルダ確認

`content/privacy.md` と `content/terms.md` は、`{appName}`、`{operatorName}`、`{lastUpdated}` 等のプレースホルダが `site.config.ts` の `legal` セクションから自動置換されます。**通常は本文の編集は不要です。**

ただし、以下の場合は本文に追記が必要です。事前に利用者に確認してください：

- 課金以外に決済を扱う場合
- 子ども向けアプリで COPPA 対応が必要な場合
- EU向けで GDPR 対応が必要な場合
- 位置情報、カメラ、連絡先などセンシティブな権限を使う場合
- 第三者にデータを提供する場合

### Step 5. 動作確認

```bash
npm install
npm run dev
```

`http://localhost:3000` で表示確認してください。ビルドが通ることも確認します：

```bash
npm run build
```

## 4. よくあるカスタマイズ依頼への対応パターン

### 「セクションを追加して」

新しいセクションが必要な場合：

1. `components/` に新コンポーネントを作成（既存コンポーネントのスタイルパターンに従う）
2. データ部分は `site.config.ts` に対応するブロックを追加し、型を export
3. `app/page.tsx` でインポートして配置

スタイル指針：
- 縦パディングは `py-24 lg:py-32`
- 中央寄せコンテナは `max-w-6xl mx-auto px-6 lg:px-8`（小さめなら `max-w-3xl`）
- セクションの見出しは `eyebrow` クラス + `font-display text-4xl md:text-5xl` の組み合わせ
- 交互の背景は `bg-[color:var(--color-surface-alt)]` を使う

### 「英語版を追加して」

Next.js の i18n routing を使う方法と、`site.config.ts` を多言語化する方法があります。MVPなら後者を推奨：`siteConfig.ja` / `siteConfig.en` のように分け、`app/[locale]/page.tsx` で切り替え。

### 「ブログ機能を追加して」

`content/blog/*.md` を作り、`lib/markdown.ts` の仕組みを流用して `app/blog/[slug]/page.tsx` を generateStaticParams で生成してください。

### 「お問い合わせフォームを実際に動かしたい」

3つの選択肢を提示してください：

1. **Formspree / Resend Forms（最も簡単）**: 登録してエンドポイントURLを `siteConfig.contact.endpoint` に貼るだけ
2. **Resend SDK + API Route**: `app/api/contact/route.ts` を作り、Resend で送信。API キーを Vercel の環境変数に設定
3. **mailto: フォールバック（現状）**: 何も設定しなくても動くが、ユーザーのメールクライアントを起動するだけ

### 「Open Graph 画像を作って」

`public/og.png` (1200x630px) を生成してください。または `app/opengraph-image.tsx` で Next.js の動的OG画像生成機能を使ってもOKです。

## 5. 必ず守ること

- **`site.config.ts` で表現できることは、コンポーネントの直接編集では行わない**
- 配色変更は CSS 変数のみ。Tailwind の色クラス（`bg-blue-500` 等）を直接使わない
- コミット前に `npm run build` でビルドエラーがないことを確認する
- 利用者の指示にない機能を勝手に追加しない（特に分析タグ、外部スクリプト、トラッキング）
- 法的ページのテンプレートはあくまで雛形であり、**実際のリリースでは法的に問題ないか専門家の確認を勧める**ことを利用者に伝える

## 6. デプロイ

利用者がカスタマイズを終えたら、Vercel へのデプロイを案内してください：

1. このプロジェクトを GitHub にプッシュ
2. [vercel.com](https://vercel.com) で「Import Project」から該当リポジトリを選択
3. ビルド設定はすべてデフォルト（Next.js の自動検出が効きます）
4. デプロイ完了後、`site.config.ts` の `app.url` を本番URLに更新して再デプロイ

カスタムドメインを使う場合は、Vercel の Dashboard > Settings > Domains から設定できることを伝えてください。
